# 全局调用与回写规则

这份规则说明：当你在其他项目里问我问题时，我应该怎么调用这个知识库；当项目里出现新的重要信息时，又应该怎么保存回这个知识库。

## 目标关系

这个知识库有两个角色：

1. 全局事实源
2. 全项目的沉淀库

它们不是两套互相独立的东西，而是同一套内容的两种用法：

- **查询时**，我先把它当事实源来查
- **整理时**，我把项目中的重要内容回写进来

也就是说，项目负责产生信息，知识库负责沉淀信息，再反过来为后续项目提供判断依据。

## 其他项目怎么调用

当你在别的项目里问我问题时，我会按这个顺序处理：

1. 先看当前项目自己的文件和上下文
2. 再查这个知识库里的相关页面
3. 把两边的信息放在一起比较
4. 如果是公司、客户、联系人、产品、市场、历史背景、固定偏好这类事实，优先以知识库为准
5. 如果是当前项目特有的实现、配置、临时决策，优先以当前项目为准

## 哪些内容要回写

下面这些内容，通常应该从项目回写到知识库：

- 公司信息
- 客户信息
- 联系人信息
- 产品信息
- 市场信息
- 长期有效的业务背景
- 会反复用到的决策依据
- 重要的项目结论
- 可以跨项目复用的流程和方法
- 新发现的冲突点或修正后的事实

下面这些内容，通常只留在项目里，不必回写：

- 一次性的实现细节
- 只对当前仓库有效的临时配置
- 很短期的试验性修改
- 与全局事实无关的局部代码细节

## 怎么保存回知识库

回写时按这个顺序做：

1. 先判断这条信息属于哪个主题
2. 放到对应的 `wiki` 页面
3. 如果没有合适页面，就新建一页
4. 更新 `index.md`
5. 必要时更新 `log.md`

## 页面放哪里

- 公司、客户、联系人这类内容，优先放到 `wiki/entities/`
- 产品、市场、业务方法这类内容，优先放到 `wiki/02-products/`、`wiki/03-markets/`、`wiki/concepts/`
- 方法、规则、长期判断这类内容，优先放到 `wiki/concepts/`
- 两个对象之间的比较，放到 `wiki/comparisons/`
- 总体说明和流程说明，放到 `wiki/overviews/`

## 冲突怎么处理

如果项目文件和知识库不一致，不要直接混着用。先判断冲突类型：

- **知识库更可信**：说明项目文件只是临时状态，按知识库走，并考虑更新项目里的说明
- **项目更可信**：说明知识库过时，先在知识库里标记需要修正，再按项目当前状态处理
- **两边都不完整**：先保留差异，等补证据后再定稿

核心原则是：冲突要显式存在，不能悄悄吞掉。

## 读写顺序

### 读

项目问题 -> 当前项目上下文 -> 知识库 -> 综合判断

### 写

项目新信息 -> 判断是否长期有用 -> 写回知识库 -> 更新目录和日志

## 简单判断标准

如果一条信息满足下面任意一条，通常值得回写：

- 以后还会再用
- 以后还会影响判断
- 以后还会给别的项目提供背景
- 以后会被客户、同事或你自己反复问到
- 会影响客户开发、产品匹配、市场判断或内容输出

如果只对当前任务有效，通常不回写。
